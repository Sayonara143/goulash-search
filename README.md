# Сервис поиска и сравнения поставщиков продуктов питания

Сервис по категории товара и региону находит поставщиков в открытых источниках,
собирает по ним структурированную карточку (контакты, условия, сертификаты),
ранжирует по релевантности и полноте данных и помогает решить, **с кем связаться в
первую очередь**.

Стек: **GigaChat structured output** для извлечения структуры, собственный
поиск + скрапинг для сбора данных, **Streamlit** для демонстрации.

---

## Идея в одном абзаце

Менеджер по закупкам тратит часы на ручной поиск: гуглит, открывает десятки сайтов,
выписывает телефоны и условия в таблицу. Сервис автоматизирует именно это:
вводишь «мука пшеничная / Москва» — получаешь готовую сравнительную таблицу
поставщиков с контактами и оценкой, по которой видно, к кому идти первым.

---

## Как это работает

Пайплайн из четырёх шагов:

```
Запрос (категория + регион)
        │
        ▼
1. Поиск      → находит страницы-кандидаты            (Yandex Search API, Tavily или DuckDuckGo)
        │
        ▼
2. Скрапинг   → качает страницы, чистит текст,         (httpx + trafilatura + Playwright fallback)
        │        вытягивает телефоны/почту регуляркой
        ▼
3. Извлечение → GigaChat раскладывает текст по JSON-схеме (response_format=json_schema)
        │        Supplier; pydantic валидирует ответ
        ▼
4. Ранжирование → дедуп + оценка по релевантности и     (детерминированная логика)
                  полноте карточки, причины оценки
        │
        ▼
Сравнительная таблица + карточки
```

Ключевой объект — модель `Supplier` (`supplier_finder/models.py`). Она и описывает
выдачу, и служит целевой схемой для GigaChat: модель обязана вернуть данные именно
в этих полях. Ответ GigaChat валидируется через pydantic, а ошибки по каждой ссылке
показываются в отладочном блоке интерфейса. Так «сырой» текст с сайтов разной вёрстки
превращается в одинаково сравнимые записи.

## Что собирается по каждому поставщику

Прямо по требованиям ТЗ:

| Поле | Источник |
|------|----------|
| Название, описание, товары | GigaChat из текста страницы |
| Сайт, телефон, email | регулярка по HTML + GigaChat |
| Регион работы | GigaChat |
| Минимальный объём заказа | GigaChat |
| Цена / ценовой ориентир | GigaChat |
| Сертификаты (ГОСТ, ТР ТС, ISO, ХАССП) | GigaChat |
| Условия доставки | GigaChat |
| Источник (URL) | пайплайн |
| Оценка + причины «почему стоит связаться» | ранжирование |

Если поля на странице нет — модель возвращает `null`, а не выдумывает. Это
осознанное правило в системном промпте: лучше пустое поле, чем недостоверное.

---

## Почему такой формат и стек (обоснование выбора)

**Почему structured output, а не просто промпт к LLM.**
Нужны не «ответы текстом», а данные, которые можно сравнивать и складывать в таблицу.
GigaChat получает `response_format=json_schema`, поэтому модель генерирует JSON по
описанной схеме, а pydantic дополнительно валидирует результат. Это убирает целый
класс ошибок «модель ответила не в том формате».

**Почему GigaChat.** Поставщики продуктов питания и их сайты — русскоязычные;
GigaChat хорошо работает с русским и подходит по требованиям к размещению данных в РФ.

**Почему собственный поиск/скрапинг, а не встроенный веб-поиск модели.**
Нативного веб-поиска (как `WebSearchTool` у OpenAI/Anthropic) у GigaChat нет — он
выполняется на стороне провайдера, а GigaChat в этот список не входит. Поэтому сбор
данных вынесен в отдельный слой (`tools.py`), что даёт контроль над источниками,
скоростью скрапинга и стоимостью.

**Почему детерминированный пайплайн, а не полностью автономный агент.**
Function-calling у GigaChat в бете и не всегда стабилен. Пайплайн предсказуем,
дешевле и контролируем по нагрузке. Скрапинг может выполняться параллельно, а
запросы к GigaChat идут строго последовательно: следующий начинается только после
ответа на предыдущий. LLM используется точечно — только на извлечении структуры,
где он реально незаменим. (Вариант с автономным агентом — см. ниже.)

**Почему Streamlit для демо.** Минимум кода — максимум наглядности: таблица,
карточки, фильтры. Достаточно, чтобы показать ценность, и легко переписать во
FastAPI-сервис для прода.

## Запуск

```bash
pip install -r requirements.txt
playwright install chromium  # нужно для сайтов, где данные подгружаются через JS
cp .env.example .env        # впишите GIGACHAT_CREDENTIALS
```

Демо без ключей (на встроенных данных — для показа UI и логики):

```bash
streamlit run app.py        # галочка «Демо-режим» включена
python cli.py "мука пшеничная" --region Москва --mock
```

Боевой режим (нужен ключ GigaChat; поиск работает через Yandex Search API,
Tavily или DuckDuckGo):

```bash
python cli.py "мука пшеничная" --region Москва
streamlit run app.py        # снять галочку «Демо-режим»
```

Тестовый режим без поисковых API:

```bash
USE_FIXED_SEARCH_URLS=true python cli.py "мясо говядина" --region Екатеринбург
```

При `USE_FIXED_SEARCH_URLS=true` сервис не обращается к Yandex Search API, Tavily
или DuckDuckGo, а берёт ссылки из `supplier_finder/fixed_search_urls.py`.

Слайдер `Сколько поставщиков` задаёт общий лимит: сервис берёт не больше этого
числа ссылок из поиска, скрапит и отправляет в GigaChat только эти страницы, а
затем обрезает финальную таблицу тем же значением. Для Yandex Search API размер
одной страницы выдачи задаётся отдельным слайдером `Ссылок с одной страницы Яндекса`.

## Docker

Локальная сборка и запуск:

```bash
docker build -t supplier-finder .
docker run --rm --env-file .env -p 8501:8501 supplier-finder
```

Production compose-файл ожидает готовый образ:

```bash
APP_IMAGE=ghcr.io/<owner>/<repo>:latest APP_PORT=8501 docker compose -f docker-compose.prod.yml up -d
```

Если для GigaChat нужен отдельный CA bundle, положите файл на сервер в `certs/`
рядом с `docker-compose.prod.yml` и укажите в `.env`, например:
`GIGACHAT_CA_BUNDLE_FILE=/app/certs/russian_trusted_root_ca.pem`.

## CI/CD на GitHub Actions

Workflow `.github/workflows/deploy.yml` при push в `main` собирает Docker-образ,
публикует его в GitHub Container Registry (`ghcr.io`) и по SSH обновляет контейнер
на Ubuntu-сервере.

На сервере должны быть установлены Docker и Docker Compose plugin. Пользователь,
под которым выполняется SSH, должен иметь право запускать `docker` и писать в
директорию деплоя, по умолчанию `/opt/supplier-finder`.

GitHub secrets:

| Secret | Назначение |
|--------|------------|
| `SSH_HOST` | IP или домен сервера |
| `SSH_USER` | SSH-пользователь на сервере |
| `SSH_KEY` | приватный SSH-ключ для деплоя |
| `SSH_PORT` | SSH-порт, опционально, по умолчанию `22` |
| `APP_ENV` | содержимое production `.env`; если не задано, `.env` должен уже лежать на сервере |
| `APP_PORT` | внешний порт приложения, опционально, по умолчанию `8501` |
| `DEPLOY_PATH` | директория деплоя, опционально, по умолчанию `/opt/supplier-finder` |
| `GHCR_USERNAME` | GitHub username для pull из GHCR, нужен для private package |
| `GHCR_TOKEN` | token с `read:packages`, нужен для private package |

Для публичного GHCR package `GHCR_USERNAME` и `GHCR_TOKEN` можно не задавать.

## Структура проекта

```
supplier_finder/
├── models.py      # схемы: Supplier, SearchRequest, RankedSupplier
├── config.py      # настройки из .env (pydantic-settings)
├── llm/
│   └── gigachat/
│       ├── client.py    # клиент GigaChat и публичные методы LLM-запросов
│       └── requests/    # реализации конкретных запросов
├── tools.py       # поиск (Yandex/Tavily/DDG) + скрапинг (httpx+trafilatura+Playwright)
├── fixed_search_urls.py  # фиксированная выдача поиска для тестов
├── agent.py       # асинхронная обёртка над LLM-извлечением
├── ranking.py     # дедуп + оценка релевантности/полноты
├── pipeline.py    # оркестрация всех шагов
└── mock_data.py   # демо-данные для запуска без API
app.py             # Streamlit UI (демо)
cli.py             # запуск из терминала
```

## Вариант с автономным агентом (для обсуждения)

Тот же скрапинг можно подключить как инструмент агента — тогда модель сама решает,
какие страницы открывать:

```python
from pydantic_ai import Agent, RunContext

agent = Agent(build_model(), output_type=ExtractedSuppliers)

@agent.tool
async def fetch_page(ctx: RunContext, url: str) -> str:
    """Скачать страницу и вернуть очищенный текст."""
    ...
```

В прототипе выбран пайплайн: он предсказуемее и дешевле при бете function-calling
у GigaChat. Автономный агент даёт больше гибкости, но менее контролируем по стоимости.

## Ограничения и куда развивать

- Сайты с тяжёлым JS рендерятся через Playwright только если быстрый HTTP-скрапинг
  не смог извлечь текст.
- Уважение `robots.txt` и rate-limiting — обязательны для продакшена.
- Кэш результатов и фоновое обновление карточек.
- Экспорт в Excel/CRM, сохранение истории поиска, заметки менеджера по поставщику.
- Дедуп по домену/ИНН вместо имени — точнее.
